# Cursor Rules for TanStack Dashboard Project

## Translation Rules

- **ALWAYS** add translations for any new text strings in both `src/i18n/en.json` and `src/i18n/de.json`
- When adding new UI components or pages, check if translations are needed
- Translation keys should be descriptive and follow the existing naming convention (e.g., `component.action`, `page.title`)
- Sort translation keys alphabetically within each file
- Use `FormattedMessage` component for all user-facing text
- Never hardcode strings in components - always use translation keys
- When adding new features, add corresponding translation keys immediately

## Code Style & Structure

- Use TypeScript for all new code
- Follow existing component patterns in `src/components/ui/`
- Use TanStack Router for routing
- Use TanStack Query for data fetching
- Follow the existing file structure and naming conventions
- Use shadcn/ui components when available

## Icon Import Convention

- **ALWAYS** import icons with the `Icon` postfix to make them immediately identifiable as icon components
- Use the pattern: `import { IconNameIcon } from 'lucide-react'`
- Examples:
  - `import { UserIcon } from 'lucide-react'`
  - `import { LockIcon } from 'lucide-react'`
  - `import { ShieldIcon } from 'lucide-react'`
  - `import { CameraIcon } from 'lucide-react'`
  - `import { UsersIcon } from 'lucide-react'`
- This makes it clear when a variable is an icon component vs other types
- Use the `Icon` suffix in variable names when destructuring: `const { UserIcon, LockIcon } = lucideReact`

## Component Guidelines

- Create reusable components in `src/components/ui/` when appropriate
- Use proper TypeScript types and interfaces
- Follow React best practices (hooks, functional components)
- Use proper error boundaries and loading states

## File Organization

- Keep related files together in feature folders
- Use index files for clean imports
- Follow the existing route structure in `src/routes/`

## Testing & Quality

- Add proper error handling for API calls
- Use proper loading states and skeleton components
- Ensure accessibility standards are met
- Test components in both light and dark themes

## API & Data

- Use the existing API patterns in `src/api/`
- Handle loading, error, and success states properly
- Use proper TypeScript types for API responses

## Git Commit Guidelines

- Write concise, specific commit messages that clearly describe the change
- Use conventional commit format: `type(scope): description`
- Keep descriptions under 50 characters when possible
- Use imperative mood ("add" not "added", "fix" not "fixed")
- Examples:
  - `feat(header): add sticky positioning`
  - `fix(scroll): prevent iOS bounce effect`
  - `refactor(layout): improve sidebar structure`
  - `docs(readme): update API configuration`
- Avoid generic messages like "update", "fix", or "improve" without context

## Remember

- This is a hotel management dashboard application
- Focus on user experience and accessibility
- Maintain consistency with existing design patterns
- Always consider internationalization (i18n) requirements
